# Governance and SBOM by construction: capa_claimdesk

A governance pack for the `capa_claimdesk` showcase. It maps every
security guarantee the program claims to the **concrete, machine-readable
evidence** the Capa compiler emits for it, and to the exact command that
checks each one.

The point of this pack: the evidence below is **generated by the
compiler, by construction**, not asserted in prose. An auditor reads the
artifacts and reruns the commands; the document only points at them.

## Honesty boundary (read first)

This is a **language showcase**, not a certified product.

- The integrity / authentication crypto is REAL: HMAC-SHA256 (the
  ledger hash chain and the keyed IBAN fingerprint) and the
  constant-time tag comparison come from `capa_hash`, a pure,
  zero-capabilities Capa library whose release is GPG-signed and
  SLSA-attested and verified by `capa install`. Those are genuine
  HMAC-SHA256 values.
- Capa is a programming language: it addresses the **technical** slice
  of any compliance regime that interacts with how a product is built
  and described. The **organisational** obligations (vulnerability
  disclosure policy, incident notification, conformity assessment,
  update distribution) are the manufacturer's regardless of language and
  are out of scope here.
- The information-flow guarantee is intra-procedural with
  whole-aggregate granularity (see the project README). It holds for the
  flows this program uses; it is not a whole-system proof.

## Artifacts in this pack

| File | What it is | Generated by |
|---|---|---|
| `manifest.json` | Per-function capability manifest + typestates + IFC declassifications | `capa --manifest main.capa` |
| `sbom.cyclonedx.json` | CycloneDX 1.5 SBOM with capability metadata (one component per function) | `capa --cyclonedx main.capa` |
| `sbom.spdx.json` | SPDX 2.3 SBOM | `capa --spdx main.capa` |
| `vex.cyclonedx.json` | VEX (exploitability statements) | `capa --vex main.capa` |
| `provenance.slsa.json` | SLSA v1 build provenance | `capa --provenance main.capa` |

All five are emitted by `governance/generate.sh`, which resolves the
`capa_claimdesk.*` self-imports by putting the repository's parent on
`CAPA_PATH` (the same invariant the main program and the negative-case
runner use).

The pack is **byte-for-byte reproducible on every OS**. To verify it:
clone the repo, run `capa install` (to vendor and verify the
dependencies), then run `./governance/generate.sh` and diff its output
against what ships here. Every file comes out **identical, with zero
differences**, build timestamps included, whether you run on Linux,
macOS, or Windows. The compiler writes these artefacts with LF newlines
through the binary stdout buffer (no platform translation), and
`.gitattributes` pins the checked-out pack to LF, so a regenerated
checkout matches the committed bytes regardless of `core.autocrlf`.

The build timestamps are pinned, not stamped at run time: `generate.sh`
reads a fixed Unix epoch from the versioned `governance/SOURCE_DATE_EPOCH`
file and exports it as `SOURCE_DATE_EPOCH` (the reproducible-builds.org
convention), so the compiler derives `timestamp`, `created`,
`annotationDate`, `startedOn`, and `finishedOn` from that instant
instead of the wall clock. The epoch is a constant in the repo,
deliberately *not* derived from the commit time (that would be circular:
the pack is generated and committed together, and an auditor checking out
the commit would observe a different HEAD time), and is bumped by hand
per release.

## Why a committed snapshot (not a gitignored build output)

This repo **commits** the five artefacts, unlike `capa_cra_template`
(which gitignores them as release assets). The reason is that this is a
*showcase*: the SBOM in the tree is part of the demonstration, and
`governance/audit.capa` reads the committed CycloneDX SBOM back in to
self-audit the program. The snapshot carries a standing promise: a
re-run of `generate.sh` leaves `git status` clean. If it ever did not,
that is a reproducibility regression and a bug to file. This is the same
committed-snapshot pattern as `capa_paymentguard`.

## Headline findings for this program

Read straight off `manifest.json` `summary`:

- **213 functions** (the program plus the verified dependencies it
  pulls in), **0 crossing `Unsafe`** (`functions_crossing_unsafe: 0`):
  no FFI / raw-pointer escape hatch anywhere in the program or its
  dependencies.
- **1 typestate protocol** (`protocol_states: 1`): the `Claim`
  lifecycle.
- **3 declassification sites** (`declassification_sites: 3`): every
  secret-to-public disclosure in the program, each with a stated reason
  (see the IFC row below).
- **26 functions declare a capability**; the remaining 187 declare
  none. The pure policy / domain core declares **zero** capabilities.

## The guarantees, mapped to evidence

Each guarantee below is enforced by the compiler. The "evidence" column
is what an auditor reads in the pack; the "check" column is the command
that re-proves it. The negative cases live under `negative/` and are run
by `bash negative/run_negative.sh`; each is a program that **must not
compile**, and the runner fails loudly if any one does.

### 1. Typestate: the claim protocol is forced

The claim moves `Draft -> Submitted -> UnderReview -> Approved/Rejected
-> Settled`; the state lives in the type, so an out-of-order transition
does not compile.

- **Evidence**: `manifest.json` -> `typestates[0]` lists the `Claim`
  protocol with its six states:

  ```
  capa --manifest main.capa | jq '.typestates[0]'
  { "name": "Claim",
    "states": ["Draft","Submitted","UnderReview","Approved","Rejected","Settled"] }
  ```

- **Negative cases** (must be rejected):
  - `typestate_skip_state.capa` -> rejected: *expects Claim[UnderReview],
    got Claim[Draft]* (cannot review a claim that was never submitted).
  - `typestate_use_after.capa` -> rejected: *consumed earlier and cannot
    be used again* (a state transition consumes the old claim value).

### 2. Linear: payment authorization is use-once

The token that authorizes a payout is minted once and consumed exactly
once: no double payment, no silently-dropped approval.

- **Evidence**: in `manifest.json`, the payout authorization carries
  `linear_obligations`; consuming functions list the token under
  `consumes` and producers set `produces_linear: true`.
- **Negative cases**:
  - `linear_double_spend.capa` -> rejected: *consumed earlier and cannot
    be used again* (a token cannot pay two claims).
  - `linear_drop.capa` -> rejected: *dropped without being consumed* (an
    approved claim cannot go unpaid).

### 3. IFC: the @secret IBAN cannot leak unmasked

The employee IBAN is `@secret`. It reaches a public sink (the ledger, a
report, stdout) only through an explicit, audited `declassify`; every
such disclosure is recorded in the SBOM with its reason.

- **Evidence**: `manifest.json` records exactly **3** declassification
  sites. The two load-bearing IBAN disclosures live in `run_claim`
  (`main.capa`), which opts into `@strict_ifc`:

  ```
  capa --manifest main.capa | jq '[.functions[] | select(.declassifications | length > 0)
        | {fn:.name, sites:.declassifications}]'
  ```

  | Function | Disclosure | Reason recorded in the SBOM |
  |---|---|---|
  | `run_claim` | `mask_iban(iban)` | GDPR data minimisation: a ledger/report shows only the last four IBAN characters, never the full account |
  | `run_claim` | `iban_fingerprint(iban, fingerprint_key())` | GDPR: a keyed one-way IBAN fingerprint correlates an employee across ledger entries; it is recorded in the audit ledger, never the IBAN itself |
  | `_capa_m13__read_cents` (`config.capa`) | `raw` | policy threshold from environment: a public config number (a cents limit), not a deployment secret |

  The first two are the IBAN-to-ledger bridge; the third declassifies a
  public config number, not regulated data. `run_claim` carries the
  `strict_ifc` attribute (`manifest.json` -> `functions[].attributes`),
  so deleting either IBAN declassify makes `capa --check main.capa` fail:
  a `@secret` value would reach `Fs.write` / `Stdio.println`.

- **Negative cases**:
  - `ifc_leak_field.capa` -> rejected: *a @secret value reaches
    Stdio.println* (printing the IBAN field directly).
  - `ifc_leak_destructure.capa` -> rejected: *a @secret value reaches
    Stdio.println* (laundering the IBAN through a destructuring bind).

### 4. Constant-time: the ledger MAC check leaks no timing

The ledger chain is verified with a constant-time MAC comparison, so a
forged chain value cannot be recovered byte-by-byte through timing.

- **Evidence**: `manifest.json` -> `mac_matches` (`ledger.capa:180`) has
  `constant_time: true` (the `@constant_time` attribute), alongside the
  `capa_hash` constant-time primitives it calls (`bytes_equal`,
  `strings_equal`):

  ```
  capa --manifest main.capa | jq '.functions[] | select(.name=="mac_matches")
        | {name, constant_time, attributes}'
  ```

- **Negative case**:
  - `ct_secret_branch.capa` -> rejected: *constant-time violation*
    (branching a constant-time function on a secret-derived condition).

### 5. Least privilege: the core is pure, the program excludes Unsafe

Every function declares only the capabilities it uses. The pure policy /
domain core (`rules`, `engine`, `mask`, `report`, `money`, `claim`,
`domain`) declares **none**, and no function in the program crosses
`Unsafe`.

- **Evidence**:
  - `manifest.json` -> `summary.functions_crossing_unsafe: 0`.
  - Per-module declared capabilities (read off `functions[].declared_capabilities`
    grouped by `pos`):

    | Module | Declared capabilities |
    |---|---|
    | `rules`, `engine`, `mask`, `report`, `money`, `claim`, `domain` | *(none: the pure core)* |
    | `config` | `Env` |
    | `fx` | `Net` |
    | `intake`, `ledger` | `Fs` |
    | `store` | `Db` |
    | `scan` | `Proc` |
    | `notify` | `Notifier`, `Stdio` |
    | `main` | all eight, attenuated at the point of use |

  - Each core function lists the capabilities it *cannot* reach under
    `provably_excluded_capabilities` (`Net`, `Fs`, `Db`, ... and
    `Unsafe`). The risk engine **cannot** exfiltrate, by type.

  ```
  capa --manifest main.capa | jq '.summary.functions_crossing_unsafe'   # 0
  capa --manifest main.capa | jq '[.functions[] | select(.declared_capabilities|length>0)
        | {fn:.name, caps:.declared_capabilities}]'
  ```

- **Self-audit** (`governance/audit.capa`): a standalone Capa program
  that reads `governance/sbom.cyclonedx.json` with the `capa_sbom`
  library and re-proves this row from the SBOM, not the source:

  ```
  CAPA_PATH=.. capa --run       governance/audit.capa
  CAPA_PATH=.. capa --wasm --run governance/audit.capa   # identical
  ```

  It checks two policies and prints PASS/FAIL: (1) the pure core declares
  no capabilities, (2) only the FX egress (`fx`, `main`) reaches `Net`.

### 6. Tamper-evident integrity: the ledger is an HMAC chain

Every lifecycle event is appended to an HMAC-chained, append-only
ledger; altering or deleting any past entry breaks every chain value
after it.

- **Evidence**: the `ledger.capa` functions (`build_log` / `append`,
  `verify_chain`, `mac_matches`) declare exactly `Fs` (write the log)
  and call `capa_hash`'s HMAC-SHA256; `verify_chain` declares **no**
  capability (it is a pure re-derivation). The runtime `out/ledger.log`
  is the chained artifact; `verify_chain` re-derives the whole chain and
  compares each stored MAC in constant time.

  ```
  capa --manifest main.capa | jq '.functions[] | select(.name=="verify_chain")
        | {name, caps:.declared_capabilities}'   # caps: []
  ```

## Regulatory mapping (honest, non-exhaustive)

Only the rows this program supplies concrete evidence for are listed.
The mapping is to the **technical** requirements a language can support;
the organisational obligations are out of scope.

| Reference | Requirement (paraphrased) | Evidence in this pack |
|---|---|---|
| CRA Annex I Part I (2)(b) | secure-by-default configuration | Every function defaults to zero capabilities; `manifest.json` shows each declared set is exactly what its signature carries (the pure core declares none). |
| CRA Annex I Part I (2)(d) | protection from unauthorised access | Capabilities are unforgeable and flow only through parameters; `manifest.json` `declared_capabilities` per function is the access-control surface. |
| CRA Annex I Part I (2)(e) | confidentiality of processed data | IFC: the IBAN is `@secret`; the compiler forbids it reaching a public sink unmasked. The 3 `declassifications` enumerate the only sanctioned disclosures, with reasons. |
| CRA Annex I Part I (2)(f) | integrity against manipulation | The ledger is an HMAC chain verified in constant time; an SBOM diff flags any dependency edit that widens a capability set. |
| CRA Annex I Part I (2)(g) | data minimisation | Least authority is the language default; the IBAN reaches the ledger only as a 4-char suffix or a keyed fingerprint, never in full. |
| CRA Annex I Part I (2)(j) | limit attack surfaces / external interfaces | The declared capability set per function *is* the interface contract; `manifest.json` and the CycloneDX SBOM are the machine-readable surface. |
| CRA Annex I Part II (1) | SBOM in a machine-readable format | `sbom.cyclonedx.json` (CycloneDX 1.5) and `sbom.spdx.json` (SPDX 2.3), each a superset of the NTIA minimum: not just what is included, but what each component *can do*. |
| GDPR Art. 5(1)(c) | data minimisation | The IBAN (personal data) is `@secret` and is disclosed only as a masked suffix or a one-way fingerprint; the two declassify sites in `run_claim` are the audited, minimal disclosures. |

Out of scope for the language layer (the manufacturer's responsibility):
the *encryption* of data at rest, DoS resilience, data erasure, and all
of CRA Annex I Part II beyond (1) (disclosure, remediation timelines,
update distribution).

## Reproducing and checking everything

```sh
capa install                               # vendor + verify dependencies
./governance/generate.sh                   # emit the 5 artefacts (byte-reproducible)
git status --porcelain governance/         # empty: the snapshot matches a fresh build

CAPA_PATH=.. capa --check main.capa        # the program type-checks clean
CAPA_PATH=.. capa --run       governance/audit.capa   # self-audit: PASS
CAPA_PATH=.. capa --wasm --run governance/audit.capa  # identical PASS
bash negative/run_negative.sh              # every guarantee rejected as expected
CAPA_PATH=.. capa test --both              # positive suite, both backends
```
